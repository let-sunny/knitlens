# Plan Format Rules

플랜 파일(루트의 `*.plan.md` 또는 `.cursor/plans/*.plan.md`)을 생성하거나 대규모로 수정할 때 **반드시** 아래 형식을 따른다.

## PRD 형태

플랜은 **제품 요구사항 문서(PRD)** 형태로 작성한다.

- **목차**를 문서 상단에 두고, 각 섹션에 앵커 링크를 제공한다.
- **필수 섹션**:
  1. **배경 및 선정 이유** — 왜 이 스펙을 우선하는지, 제품/아키텍처 문서와의 정합성
  2. **범위** — 이번 스펙에 포함되는 것 / 제외되는 것(후속 스펙)을 명시
  3. **사용자 스토리와 시나리오** — 사용자 관점 시나리오와 예외/에러 케이스
  4. **기능 요구사항** — 화면·API·데이터·플로우별 요구사항
  5. **비기능 요구사항 및 제약** — 성능, 보안, 호환성 등 (해당 시)
  6. **수용 기준** — QA·리뷰 시 확인할 체크리스트
  7. **기술 설계 및 데이터 흐름** — 아키텍처, DB/API 설계, 의존성
  8. **구현 계획 (TODO)** — 아래 TODO 규칙 적용
  9. **참조 문서** — `docs/`, `.cursor/` 등 관련 문서 링크
  10. **릴리즈 노트 초안** — 아래 체인지로그 규칙 적용

문서 상단 **frontmatter**에는 `name`, `overview`, `todos`, `isProject` 등을 포함하고, `todos`는 구현 계획 섹션과 동기화한다.

## TODO 상세화

구현 계획(TODO)은 **상세히** 적는다.

- 각 TODO 항목은 **한 문장으로 요약 가능한 단위**이되, **구체적인 산출물·위치**를 포함한다.
  - 예: "Supabase 프로젝트 연결, projects/pattern_sources 테이블, Storage 버킷 및 env 설정"
  - 예: "POST /api/pattern/analyze 구현 (LLM, 스키마 검증, 상태 저장)"
- **순서**는 의존 관계를 반영한다 (인프라 → CRUD → API → 화면 등).
- **ID**는 kebab-case로 짓고, frontmatter `todos`와 본문 목록이 **일치**하도록 유지한다.
- 필요 시 하위 단계(sub-bullet)로 파일 경로, API 시그니처, 검증 조건을 적는다.

## 체인지로그(릴리즈 노트)

플랜 문서 **끝**에 **릴리즈 노트 초안** 섹션을 둔다.

- 제목: `## 릴리즈 노트 초안` (또는 `## Changelog`).
- **Added** / **Changed** / **Fixed** 등으로 구분하여, 이 스펙 완료 시 사용자·개발자가 보게 될 변경 사항을 나열한다.
- 사용자 관점(새 화면, 새 동작)과 기술적 변경(새 API, DB 스키마, env)을 모두 포함해도 된다.

이렇게 하면 플랜이 단순 작업 목록이 아니라 **요구사항·수용 기준·구현·릴리즈**까지 한 문서에서 추적 가능한 PRD 역할을 한다.
